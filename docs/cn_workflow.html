Chinese localization pipeline

One command per video. Two Bilibili uploads produced in the same batch and released on a schedule — 英配中字 subs and 中配 dub, built from one shared translation. A human touches it twice: an optional title swap, and the upload + schedule pass. Everything between is deterministic file names, loud failures, and idempotent re-runs.

01

The shape

Status answers one question — whose turn is it. Machine steps (translate, render, mux) never get a status; they get a line in the run log.

🤖 machine runs → 🧑 Reviewer: review draft machine's done, upload + schedule happen under this one status → Scheduled / Done

Blocked is a branch, not a step. It only fires when pre-flight actually fails — a human owes an asset before the machine can even start. It's never a resting state for "machine finished, waiting on the human." That's just Reviewer: review draft: the files are ready, it's the human's turn to upload and schedule. Don't leave a row on Blocked once the machine work is actually done — that reads as something being wrong when nothing is.

Why the dub costs almost nothing extra. Translation happens once, in Stage 3. V2's audio is generated from that exact approved SRT — it isn't a second translation pass. That's what makes producing both versions in one batch practical: they're two renders of one translated script, not two separate products.

No formal review gate for Chinese. The team is fluent, so review happens naturally while preparing the upload — no dedicated tooling needed. A Frame.io-based native-speaker review stage is designed and partly built for the languages nobody on the team can personally judge (Spanish, Japanese, German) but isn't wired into this pipeline yet — see Open Items.

Page content, in this order — everything below is a heading block on the page itself, not a property: # CN title (V1/V2 subsections, each the exact ready-to-paste Bilibili title — brand suffix in 【 】, e.g. 【英配中字】 — plus the losing alternates, each tagged 【中配】, for reference) → # CN thumbnail (the actual image, embedded — title and cover sit together since they're the packaging) → # CN tags (plain comma list — not a multi-select, so it's never blocked by a missing option) → # CN description (main content description → sponsor CTA if flagged → 制作人员 credits, in a code fence) → # CN ad disclosure (own section, only present when Contains ads? is ticked — never inside the description fence, so it can't get pasted into Bilibili's actual description field by mistake). Matches the "Add Chinese section" skill's shape on the English pages — one convention, not two.

Thumbnail placeholder convention. When a row doesn't have a thumbnail yet, the # CN thumbnail block holds a literal [ not yet generated ] instead of being left empty or omitted — so a glance at the page tells you what's missing without opening Drive.

Kept deliberately lean: no per-version performance-metrics table. Both versions are treated as two legitimate deliverables from one shared script, not competing variants to be scored against each other — so there's nothing to track beyond "is it live, and where."

04

The pipeline

Trigger: localize {project-id} for Chinese · Batch: everything eligible that isn't yet in the Chinese DB. Idempotent — a done stage says SKIP_OK; only --force redoes it.

--force is scoped, not global. --force=<stage> reruns that stage and every stage downstream that consumes its output — e.g. forcing Stage 4 (Chinese audio) also reruns Stage 5 (render), since render consumes the dub. It never reruns upstream or sibling stages: fixing a Stage 5 render issue doesn't retranslate Stage 3 or regenerate Stage 2's title options. This matters because Stage 2 and Stage 3 are LLM calls; a global --force would silently re-spend on outputs that were already approved.

Stage 2 runs alongside Stages 3–5, not after them. Thumbnail generation depends on nothing but the thumbnail source image and the winning title — both already in hand right after pre-flight. It never touches the transcript, the translation, or the dub. Kick it off in parallel with Stage 3 instead of queuing it first or waiting for the whole audio chain; the only place the two tracks need to be both-done is Stage 6 (Bilibili upload needs the cover file and the rendered videos together). The thumbnail track (nano-banana clean + a PIL render, a few minutes) is shorter than the audio track (transcribe → translate → dub → render, 15–30 minutes) — run in parallel, it's effectively free.

1🚦 Pre-flight — the loud failure gate🤖 machine

Checks the things that most commonly break this stage. Any miss = stop, write the missing list to Status: Blocked + run log, name the owner, exit. No partial runs.

  • {id}_master.mp4 exists
  • {id}_en.srt exists — not a stop condition. If missing, generate one (pull English captions from the platform, or split it back out of an existing {id}_bilingual_ensub.srt if one's already there) and continue. Only stop if no English source can be found or built at all.
  • Thumbnail source exists (text-free preferred; baked-text is fine — Stage 2 recovers it)
  • Winning title present — if not, use the live published title, tick Title provisional?, print it loud
  • Sponsor content detected (grep the SRT against the DB's sponsor field + known promo-code patterns) — not a stop condition, just tick Contains ads? so the upload step applies Bilibili's ad-disclosure tag
  • {id}_me.wav in the project root — not a stop condition either way. Present → Stage 4 mixes it in. Absent → Stage 4 just skips the background bed, no fallback attempt.
2🎨 Meta pack + thumbnail🤖 machine

Generates 3 CN title options and picks one (writes the other two as alternates), formats the winner into the two ready-to-paste upload titles, a CN description with the 制作人员 credits block, and Bilibili tags. All glossary-checked, sentence case, under 80 chars.

Two ready-to-paste titles, not one generic one. # CN title holds two subsections, already formatted exactly as they go into Bilibili — the human editor grabs and pastes, no assembly required at upload time:

# CN title
V1 · 英配中字{中文标题}（English title）【英配中字】
V2 · 中配{中文标题} 【中配】 — not 中配中字 (redundant once the audio is Chinese)
Alternatesthe 2 losing title options, each tagged 【中配】 so they're still pasteable as a backup without reformatting

Titles must differ, or two near-identical uploads read as 搬运 (re-upload spam) — that's why V1 carries the bilingual form and V2 doesn't. Generating both here means nobody has to remember this rule at schedule time.

If Contains ads? is ticked, the sponsor CTA goes in the description — same rule as the video. Nothing gets cut from the sponsor segment when localizing, so the description can't quietly drop the link either. Translate the exact CTA line the sponsor required for the English upload (link, discount code, and all). It's never the first line — put it in the middle of the description, after the main content paragraph: main content → CTA → 制作人员 credits. A description that opens with a promo link reads as spam and buries the actual video description. The Contains ads? tag flags the upload for platform disclosure; it doesn't substitute for the actual link being present.

Also write a # CN ad disclosure section — its own heading, not a line inside the description — when Contains ads? is ticked. Bilibili's ad-disclosure step needs 内容、口播、简介 all selected — the sponsor appears in the video, gets a spoken mention, and has a link in the description, so all three apply. Keeping it as a separate section means the uploader sees exactly what to tick without leaving the Notion page, and it can never get copy-pasted into Bilibili's actual description field by mistake — a real risk if it lived inside the # CN description code fence.

Thumbnail — the hybrid approach. A text-to-image model cleans baked-in English text and rebuilds the background behind it (pixel-editing tends to ghost here); the Chinese text itself is then rendered deterministically with the brand font on the cleaned base. Don't ask the model to render the Chinese text directly — it's unreliable with CJK strokes.

Safe to auto-pick the title. Bilibili lets you edit title, cover and 简介 after publish. The video file is the one thing you can't swap — that's where the human gate goes, not here.

3📝 Subtitles — translated outside CapCut🤖 machine

CapCut has no line-to-line memory, which is how a term can quietly drift into something else 40 lines later. Translate {id}_en.srt against the full glossary, in context, as one document.

  • Max 2 lines/cue · CN ≈16 chars/line · EN ≈42 · min 1s, max 6s · breaks at clause boundaries
  • Build {id}_bilingual_ensub.srt: one plain srt, CN first line then EN second line in the same cue, same style — no separate styling layer needed. Cue timestamps here come from the English source, so they're only valid for Stage 5's _ensub.mp4 burn. See Stage 4 for why the dub needs its own re-timed copy.

4🎙️ Chinese audio — from the approved SRT🤖 machine

Generated from {id}_zh.srt, not auto-translate-and-speak — the voice and the subs must say the same words. This is also why the dub is cheap: the translation already happened in Stage 3.

Generate chunks concurrently, not in a for-loop. Each chunk is an independent TTS call — nothing about chunk 5 depends on chunk 4 having finished. A sequential run of 10–12 chunks burns minutes waiting on API round-trips for no reason; firing a handful off at once (respecting whatever rate limit applies) turns that into well under a minute for a typical project.

Don't stretch per cue. Per-cue TTS with per-cue timestamp-fitting causes audible speed-whiplash — short lines rushed, long lines dragged — and tone resets between every cue. What holds up: generate in multi-cue chunks as one continuous take, then apply a single uniform tempo stretch to the whole chunk to land it in its time window. One continuous take is also the only real lever for tone continuity — there's no cross-request context-chaining to lean on.

Critical — a chunk that overflows its window doesn't just affect that chunk. Chunks get concatenated sequentially, not anchored to absolute timeline positions — so if one chunk hits the stretch cap and still doesn't fit, the excess shifts every chunk after it later, and compounds further at any later chunk that also caps out. This isn't hypothetical: it caused real, user-reported audio/video desync in production (starting inside the first offending chunk, then persisting for the rest of the video) — caught only after the full video was already rendered, meaning a second full render was needed just to fix sync. That's the expensive way to find out, and it's avoidable entirely at the audio-assembly stage, before render, where fixes are cheap:

  • Watch for at-risk chunks before they become a problem: lots of short, punchy lines (reactions, tier/grade callouts, one-word beats) accumulate disproportionate pause overhead relative to their target window, compared to chunks with fewer, longer sentences. These are the ones likely to hit the stretch cap.
  • If a chunk still overflows after the uniform stretch (even at the cap), re-split just that chunk into smaller sub-chunks (e.g. 5 lines instead of 15) and regenerate only those pieces — a handful of extra API calls, not the whole video. Cap this at one re-split attempt. Don't loop re-split → regenerate → re-check until it happens to pass — a structurally awkward line (one short punchy beat filling a long window) can keep missing tolerance indefinitely, and each miss is a paid TTS call. One re-split is normally enough to remove most of the disproportionate pause overhead; whatever gap remains is what the corrective tempo pass below is for.
  • Mandatory gate before assembling the master track: every chunk must land within ~50ms of its target duration. After the one re-split attempt, apply a final corrective tempo pass on the whole chunk unconditionally — not as a second thing to try only if the retry also fails, but as the guaranteed close-out step every time. It's a local ffmpeg call, not an API call, so it's nearly free — a little more speed distortion beats either compounding drift or an open-ended retry loop.
  • Exception: the last chunk in the video doesn't get force-stretched. The corrective pass exists to stop a chunk's overflow from shifting every chunk after it — there's nothing after the last one, so that rationale doesn't apply. If the final chunk undershoots its window because the source cue spans a real outro pause or fade (check the actual gap — a cue with several seconds more window than its neighbors, right at the video's end, is the tell), pad the shortfall with trailing silence instead of stretching the speech to fill it. Found in production: forcing an exact fit there would have meant slowing Chinese speech ~32% to cover 8+ seconds of dead air the English original didn't fill with speech either — audible and pointless, since nothing downstream needed protecting.
  • Never render the full video until every chunk passes this fit check. Verifying chunk-by-chunk while it's still audio-only is fast and nearly free; discovering a sync bug in the rendered 4K output means paying for a full re-render to fix it.

Critical — chunk-level fit doesn't guarantee cue-level fit. The fit gate above only checks that a chunk's total duration matches its target window. It says nothing about where each individual cue lands inside that chunk. Cue timestamps come from proportionally splitting the English segment by character count — a good enough guess for English, but Chinese TTS doesn't pace a continuous take the same way the English original did internally. Found in production: a 40s chunk that hit its total-duration target inside 2ms was still several seconds off cue-by-cue by the middle of the take — audio that sounds "in sync" at the chunk boundaries but visibly drags behind the burned subtitles in between. This is a distinct failure mode from the cascading-overflow bug above; passing the chunk-duration gate does not rule it out.

  • Forced-align each chunk after generation, before building the dub-facing subtitle file: transcribe the chunk's actual TTS audio (word-level timestamps), then align that transcript against the known reference text (the exact lines sent to the TTS call — no guessing needed, order is preserved) to recover each cue's real start time inside the chunk.
  • This produces a second, dub-specific timing track: {id}_bilingual_cndub.srt, same CN+EN text as {id}_bilingual_ensub.srt but re-timed to the actual dub audio. The two are expected to diverge — one is English-timed for _ensub.mp4, the other is dub-audio-timed for _cndub.mp4. Don't burn the English-timed file onto the dub video; that's the bug.
  • Add this alignment check to the same pre-render gate as the duration fit check — both are cheap, audio-only verifications that belong before Stage 5, not discoveries made after a 4K render.
  • Run a monotonic-clamp pass on the aligned cues before writing the srt. The aligner occasionally misplaces a very short or single-character line (an isolated "Pulling." or "拉。") and produces a cue that starts before the previous one ends. Clamp every cue's start to at least the prior cue's end — cheap, and catches the whole failure class in one pass instead of chasing individual overlaps.

Background bed. {id}_me.wav present → mix it in cleanly. Absent → proceed without one, dub audio only. No fallback attempt to reconstruct M&E from the mixed master.

When it passes, save the stem as {id}_me.wav at the project root and continue normally; dub mix-me (Stage 4) mixes it into the tightened VO before render. Staff-prepped {id}_me.wav is still preferred when it already exists — this is a fallback for videos that don't have one, not a replacement for the real thing.

5⚙️ Render — ffmpeg, no editor opens🤖 machine
  • {id}_ensub.mp4 = master + _bilingual_ensub.srt burned
  • {id}_cndub.mp4 = master video + Chinese audio + _bilingual_cndub.srt burned (the forced-aligned copy — not the English-timed one)

Env note. Subtitle burn-in needs a libass-enabled ffmpeg build. The stock build on some systems silently lacks the subtitles filter — worth a one-time check, not a per-run one.

Rendering ensub and cndub concurrently is unverified, not ruled out. They share nothing but the source file, which makes them look like an obvious parallelization win — but they also share the same hardware encoder, and it's untested whether it handles two concurrent sessions cleanly or just serializes them under the hood. Validate on the actual machine before making this the default; if the encoder doesn't really run both at once, parallelizing just adds resource contention for no speedup.

Both files verified (durations match source/dub audio) → hand off to Stage 6.

6📤 Upload — stays human🧑 human · until Bilibili API access

Browser automation can't push a large video file to Bilibili (tool size caps, and native file pickers are invisible to automation). Until an official upload API is in place, a person pulls the file + cover from /CN/ and uploads both by hand — this doubles as the review pass, since the uploader is watching the draft before submitting it. The pipeline fills everything else — the ready-to-paste V1/V2 titles from Stage 2, 简介, tags, 分区, 合集 — and writes the publish kit. Both versions go up as drafts, submitted straight to scheduling, not a separate review pass.

Paste the description as-is — it's the ready-to-publish text, not a draft to edit. If the source video carries a sponsor CTA, the Chinese description on the page already has it, positioned correctly. Grabbing a shortened or "cleaned up" version at upload time risks dropping the sponsor link — a compliance problem, not just a formatting one. Check # CN ad disclosure too when it's present — it names the exact Bilibili checkboxes to tick, separately from the description so it never gets pasted into the video's actual description field by mistake.

7📅 Schedule — one click, then the machine takes over🧑 human schedules · 🤖 machine writes back

Both versions are set to publish on Bilibili's built-in scheduler rather than clicked live manually — same batch in, same batch out. Titles are already prepared in # CN title from Stage 2 — paste V1/V2 as-is, nothing left to construct here.

The write-back doesn't need a human either. Once both are scheduled, the machine reads each draft's studio page — that's just reading rendered text, not the native file-picker upload that keeps Stage 6 on a human — and writes V1/V2 published, the BV link, and the scheduled date straight into the Chinese DB row. The schedule click itself is the only thing that stays manual here.

05

Rules that keep it from rotting

  • 1One translation, one QC pass, two videos. The moment the dub gets its own translation, you have two products and double the errors.
  • 2Every glossary miss goes back into the glossary. Otherwise QC volume never drops.
  • 3Loud failures. A silent partial run that ships a wrong title is worse than a crash.
  • 4Provisional is fine. Invented is not. Missing title → fall back and flag. Never make one up quietly.
  • 5Sponsor content isn't a blocker — it's a tag. Detect it, disclose it on upload. No need to strip or delay a video over it.
  • 6The tag discloses the ad; the description still needs the actual link. The video isn't cut for localization, so nothing about the sponsor CTA should quietly disappear in the Chinese description either.
